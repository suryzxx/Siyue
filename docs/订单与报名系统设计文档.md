# 思悦教育 - 订单与报名系统设计文档

## 一、业务背景与需求分析

### 1.1 班级学期结构

| 学期 | 类型 | 特点 | 课程安排 |
|------|------|------|---------|
| 春季班 | 长学期 | 约16-18讲，4-5个月 | 每周1-2次课 |
| 暑假班 | 短学期 | 约10-12讲，1-2个月 | 每周2-3次课 |
| 秋季班 | 长学期 | 约16-18讲，4-5个月 | 每周1-2次课，可分期（秋上/秋下） |
| 寒假班 | 短学期 | 约8-10讲，2-3周 | 连续上课 |

### 1.2 三种核心业务场景

#### 场景一：正常报名
- 学生首次报名进入班级
- 支付全款后即可入学
- 名额实时占用

#### 场景二：续报（提前报名下一学期）
- 学生在春季提前报名暑假和秋季
- **暑假班**：必须付全款
- **秋季班**：可选择整期支付或分期支付
  - 秋上：前3-4讲，约25%费用
  - 秋下：后续课程，约75%费用
- 续报时给予一定优惠

#### 场景三：预售
- 新班级提前开放预售
- 学生先付定金锁定名额
- 定金人数达到开班阈值后，班级正式开班
- 未达到阈值则取消班级，退还定金

## 二、用户疑问解答

### 2.1 待支付订单处理方式

**方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 方案A：限时支付（30分钟） | 不占名额，公平 | 用户体验稍差 | 热门班级、名额紧张 |
| 方案B：不限时支付 | 用户体验好 | 可能占名额不付费 | 名额充足、预售阶段 |

**推荐方案：混合模式**
```
1. 预售阶段：不限时支付，定金即可锁定名额
2. 正式开售后：
   - 热门班级：30分钟限时支付
   - 普通班级：24小时限时支付
3. 续报优惠期：48小时限时支付（给老学员更多时间）
```

### 2.2 秋下续费操作流程

**推荐方案：新订单模式（方案1）**

原因：
- 业务逻辑清晰，易于理解和维护
- 便于财务对账和统计分析
- 保留完整的支付记录

**微信小程序（家长端）操作流程：**
```
1. 进入"我的课程"
2. 找到秋季班课程
3. 点击"续费秋下"
4. 系统显示秋下费用和优惠
5. 确认支付
6. 生成新订单，状态"待支付"
7. 完成支付后，订单状态变更为"已支付"
8. 课程表中自动显示秋下课程
```

**后台管理（管理员端）操作流程：**
```
1. 进入学生详情页
2. 查看学生的班级记录
3. 点击"办理续费"
4. 选择续费类型（秋下）
5. 系统自动计算费用
6. 选择支付方式（现金/微信等）
7. 确认后生成订单
8. 订单状态显示为"已支付"
```

### 2.3 订单状态设计

**扩展后的订单状态：**

```typescript
type OrderStatus = 
  // 基础状态
  | '待支付'           // 订单创建，等待支付
  | '已支付'           // 支付完成
  | '已取消'           // 订单取消
  | '已退款'           // 全额退款
  
  // 预售相关
  | '待付定金'         // 预售班，等待付定金
  | '已付定金'         // 定金已付，等待付尾款
  | '待付尾款'         // 班级已开班，等待付尾款
  | '预售失败'         // 未达到开班人数，预售取消
  
  // 分期相关
  | '部分支付'         // 已付秋上，待付秋下
  | '待续费'           // 秋上结束，提醒续费秋下
  
  // 退款相关
  | '部分退款';        // 部分退款
```

**状态流转图：**

```
正常报名流程：
待支付 → 已支付 → (可能) 已退款/部分退款
    ↓
  已取消（超时）

预售流程：
待付定金 → 已付定金 → 待付尾款 → 已支付
              ↓
          预售失败（自动退定金）

分期流程（秋季班）：
待支付 → 已支付（秋上）→ 待续费 → 已支付（秋下）
                        ↓
                    部分支付（中间状态）
```

#### 3.2.3 不付尾款的处理

```
情况1：班级已开班，学生不付尾款
  → 系统自动发送提醒（第1天、第3天、第7天）
  → 7天后仍不付：管理员可选择
     - 延长期限（再给7天）
     - 取消订单，退还定金（扣除手续费）
     - 转为欠费状态，继续催缴

情况2：班级未开班（预售失败）
  → 系统自动退还定金
  → 发送退款通知
  → 订单状态变更为"预售失败"
```

### 3.4 续报流程详细设计

#### 3.4.1 续报时间窗口

```
[春季进行中] ──────→ [春季结束]
     ↓                    ↓
[春季续报期开启]    [暑假开始]
     ↓                    ↓
可续报暑假、秋季     [暑假续报期开启]
                          ↓
                     可续报秋季
```

#### 3.4.2 秋季班分期支付详细流程

```
Step 1: 学生报名秋季班
  └─ 选择"分期支付"
  └─ 生成订单1：秋上（25%费用），状态"待支付"
  └─ 同时预生成订单2：秋下（75%费用），状态"待激活"

Step 2: 支付秋上
  └─ 订单1状态变更为"已支付"
  └─ 学生进入班级，开始上课
  └─ 订单2状态保持"待激活"

Step 3: 秋上结束（第3-4讲后）
  └─ 系统激活订单2
  └─ 订单2状态变更为"待支付"
  └─ 推送续费提醒

Step 4: 支付秋下
  └─ 订单2状态变更为"已支付"
  └─ 学生继续上课

Step 5: 如果不付秋下
  └─ 第1天：推送提醒
  └─ 第3天：推送提醒 + 短信
  └─ 第7天：管理员介入处理
```

## 八、技术实现建议

### 8.1 定时任务
```
1. 订单超时检查（每分钟）
   - 扫描即将超时的订单
   - 发送提醒或自动取消

2. 预售状态检查（每小时）
   - 检查预售截止时间
   - 未达标的自动退款

3. 续费提醒（每天上午10点）
   - 检查待续费订单
   - 发送提醒通知
```

### 8.2 事务处理
- 支付成功后：更新订单状态、更新班级人数、发送通知
- 预售开班：批量更新订单状态、批量发送通知
- 预售失败：批量退款、更新订单状态

### 8.3 并发控制
- 班级名额操作需加锁
- 支付回调需幂等处理
- 预售人数统计需原子操作

## 十、技术实现参考（行业最佳实践）

### 10.1 订单超时处理方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **定时任务扫表** | 实现简单，成本低 | 时间不精确，数据库压力大 | 小规模系统 |
| **JDK延迟队列** | 无外部依赖 | 内存有限，重启丢数据 | 单机简单场景 |
| **Redis过期通知** | 支持分布式 | 可能有延迟 | 中等规模 |
| **RocketMQ延迟消息** | 精确可靠 | 成本较高，需配置 | 大规模高并发 |
| **时间轮算法** | 高性能 | 实现复杂 | 极高并发 |

**推荐方案：Redis + 定时任务混合**
- Redis设置过期时间作为主通道
- 定时任务每5分钟扫描作为兜底
- 分布式锁防止重复处理

### 10.2 幽灵支付处理

对于支付回调延迟导致的「幽灵支付」（订单关闭后支付成功）：

```
支付回调时检查订单状态：
1. 如果订单已关闭，查询支付渠道真实状态
2. 如果确认支付成功：
   - 方案A：走退款流程（推荐）
   - 方案B：联系用户确认是否补发课程
3. 记录日志便于排查
```

### 10.3 微信「教育续费通」方案

微信支付提供的教育续费产品能力：
- 扣款前72小时通知用户
- 用户可自主解绑
- 每月仅扣款成功1次

**续费通知时机建议：**
- 到期前30/14/7/3/1天发送提醒
- 渠道：APP推送 + 短信 + 微信服务号

### 10.4 关键设计原则

1. **幂等性**：支付回调必须支持幂等处理
2. **最终一致性**：使用消息队列保证状态同步
3. **超时兜底**：Redis + 定时任务双重保障
4. **资金安全**：大额订单增加人工确认环节
5. **可追溯**：所有状态变更记录操作日志
